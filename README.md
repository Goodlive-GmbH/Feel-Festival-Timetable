# sv

Everything you need to build a Svelte project, powered by [`sv`](https://github.com/sveltejs/cli).

## Creating a project!

If you're seeing this, you've probably already done this step. Congrats!

```sh
# create a new project
npx sv create my-app
```

To recreate this project with the same configuration:

```sh
# recreate this project
npx sv@0.12.7 create --template minimal --types ts --add tailwindcss="plugins:typography,forms" prettier eslint --install npm .
```

## Developing

Once you've created a project and installed dependencies with `npm install` (or `pnpm install` or `yarn`), start a development server:

```sh
npm run dev

# or start the server and open the app in a new browser tab
npm run dev -- --open
```

## Building

To create a production version of your app:

```sh
npm run build
```

You can preview the production build with `npm run preview`.

> To deploy your app, you may need to install an [adapter](https://svelte.dev/docs/kit/adapters) for your target environment.

## Environment variables

| Variable                      | Values               | Default                                       | Description                                                                                                                                                                                                                    |
| ----------------------------- | -------------------- | --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `PUBLIC_SCHEDULE_IMPORT_MODE` | `dragdrop` \| `file` | `file` (production), `dragdrop` (development) | Controls how schedule data is loaded. `dragdrop` enables drag-and-drop Excel upload in the browser (intended for preview/dev only). `file` uses the pre-converted JSON from `src/lib/mockData.json` (intended for production). |

The defaults are set in `.env.development` and `.env.production`. Do **not**
enable `dragdrop` in production. For Vercel preview deployments, set this
variable in the Vercel dashboard (Project Settings → Environment Variables) to
`dragdrop`.

## Updating the schedule data

The app's seed data lives in `src/lib/mockData.json` and is generated from an
Excel timetable. To regenerate it after editing the spreadsheet:

1. Place the updated Excel file at `import/timetable.xlsx` (the "Worksheet"
   sheet, rows starting at row 4, columns: tag, von, bis, name,
   teilnehmende, gruppen, bereich, jobs, beschreibung, ereignisart, genre, typ).
2. Run the conversion script:

   ```sh
   npm run update-data
   ```

   Or with explicit paths:

   ```sh
   npx tsx scripts/convert_timetable.ts path/to/input.xlsx path/to/output.json
   ```

The script bumps the seed version (so clients reseed their local database on
next load) and **preserves** the festival-map layout — stage coordinates
(`x`/`y`) and the `pois` list are carried over from the existing
`mockData.json` rather than derived from the sheet. Commit the regenerated
`src/lib/mockData.json` afterwards.

## Manually editing `mockData.json`

`src/lib/mockData.json` is a single-element array wrapping one object with the
keys `seedVersion`, `stages`, `pois`, `artists`, and `performances`. Most of it
is regenerated by `npm run update-data`, but a few fields are **not** derived
from the Excel sheet and must be maintained by hand directly in the JSON.

Any time you hand-edit the file, bump `seedVersion` to a higher number so
clients reseed their local database on next load (the script uses `Date.now()`;
manually just use any larger integer).

### Fields the script generates (don't hand-edit — changes are overwritten)

- **`stages`** — `id`, `name`, `color` (except `x`/`y`, see below).
- **`artists`** — `id`, `name`, `type`, `genre`.
- **`performances`** — `id`, `stageId`, `artistId`, `startTime`, `endTime`,
  `type`, `info`.

### Fields that are maintained by hand (safe to edit)

These are preserved across regeneration, so this is where manual edits belong:

- **Stage map coordinates** — the `x` and `y` on each stage. Values are
  `0`–`1`, relative to the map overlay (`x` = left→right, `y` = top→bottom).
  A stage keeps its coordinates across regeneration as long as its `id` is
  unchanged; a newly added stage will have no coordinates until you add them.

  ```json
  { "id": "stage-main", "name": "Main Stage", "color": "bg-fest-blue", "x": 0.5, "y": 0.28 }
  ```

- **`pois`** (points of interest) — the entire list is hand-maintained and is
  never generated from the sheet. Each entry:

  ```json
  { "id": "poi-food-1", "name": "Food Court", "type": "food", "x": 0.55, "y": 0.66 }
  ```

  - `id` — unique, stable string.
  - `name` — display label.
  - `type` — one of `food`, `wc`, `entrance`, `medical`, `camping`, `water`,
    `info` (see `PoiType` in `src/lib/types.ts`).
  - `x` / `y` — map coordinates, same `0`–`1` convention as stages.

- **`isFavorite`** on an artist is optional and normally set at runtime; leave
  it out of the seed unless you intend an artist to be favourited by default.

> Tip: keep `id` values stable. Stage coordinates are re-matched by `id` on
> regeneration, so renaming or changing an `id` will drop its saved `x`/`y`.
